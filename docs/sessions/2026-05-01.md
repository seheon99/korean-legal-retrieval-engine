# Session 2026-05-01 — ADR-011 retention; first code (fetch script)

## Plan

Two-thread session — close the last open ADR dependency from

Transitions the project from "all-design" to "first-implementation."

### Thread 1: ADR-011 — raw-API-XML retention (Proposed → Accepted via §5b)

**Why**: ADR-008's recommendation explicitly leans on raw-API-XML
retention as the canonical fallback for un-promoted fields. If we
start coding ingestion under the "no JSONB" assumption and then
discover retention can't be committed, the schema gets ripped open
mid-implementation. Surface and close before any ingestion code
lands.

**Decision shape** (to be detailed in the ADR, not here):

- Where does raw API XML live? (filesystem under `data/raw/` vs
  `legal_documents.raw_xml TEXT` column vs object store)
- Layout / naming convention (per-doc, per-version, MST-keyed)
- Lifecycle — kept indefinitely, rotated, or pruned when superseded?
- Backup / DR posture (does it need to survive `pg_dump` only, or
  separate backup track?)
- Relation to `docs/api-samples/` (gitignored, fetched on-demand
  on Fedora). Is `data/raw/` a renamed/promoted version of that, or
  a parallel store?

Same decision-shape as ADR-006 — representational choice for a
known design slot. Should run quickly.

**§5b loop**:
1. Draft `docs/decisions/ADR-011-<slug>.md` as Proposed
2. Surface for Seheon's approval
3. On approval, flip to Accepted, apply consequent doc edits
   (CLAUDE.md §4 reference to retention, possibly ADR-008
   forward-pointer if retention shape ratifies vs triggers fallback)

**Triggers fallback if**: ADR-011 ends up rejecting durable
retention (e.g., decides the corpus is too large to keep). In that
case, ADR-008 fallback Option D activates — named-field allowlist
JSONB on `legal_documents`. ADR-008 would need a follow-up ADR or
a status revision.

### Thread 2: `scripts/fetch_law_samples.sh`

**Why**: First piece of code in the project that doesn't depend
on a new ADR. Overdue — flagged in the 2026-04-26 session log as
the natural follow-up after the Fedora handoff. Converts the
`README.md` curl block into a versioned, idempotent,
parameterizable utility.

**Deliverable**: bash script under `scripts/` that:

- Reads `LAW_GO_KR_OC` from environment (errors clearly if unset)
- Accepts target spec (Act MST, Decree MST, search query) as
  arguments — no hard-coded MSTs
- Writes XML responses to `docs/api-samples/` with deterministic
  filenames (`law-{MST}-{slug}.xml`, `search-{query}.xml`)
- Skips re-download if the file exists and `--force` is not set
  (idempotent)
- Exits non-zero on HTTP error or empty response
- Single dependency: `curl` (already on Fedora)

**Default invocation** (zero-arg) reproduces the three Phase-1
samples (`MST=228817` Act, `MST=277417` Decree, `query=중대재해`
search). Matches the convention in `README.md`.

**Out of scope for the script**:
- XML validation / parsing — that lives in the ingestion pipeline,
  not the fetcher
- DB writes — fetcher just persists files
- Retention policy enforcement — that's ADR-011's territory

## What happened

Two-thread session executed as planned.

1. **ADR-011 accepted — raw-API-XML retention as a filesystem store.**
   Closes ADR-008's soft retention dependency (Trade-offs §1, Out of
   scope §4, Revisit triggers §1). Decision: filesystem at
   `data/raw/{law_id}/{mst}.xml`, indefinite retention, plain UTF-8,
   gitignored. Integrity link to the database row is the existing
   `legal_documents.content_hash` (ADR-010 frozen column). Atomic
   writes via tmp + rename. Five options enumerated (filesystem,
   DB column, object store, no retention, gzip); A proposed and
   accepted; B held as steelman, C as premature, D as
   ADR-008-fallback trigger, E as Phase-2 revisit. ADR draft drew
   no patches before approval. Commit `bbfecc8`.

2. **First code in the project: `scripts/fetch_law_samples.sh`.**
   Bash script that converts the prior README curl block into a
   versioned, idempotent, parameterizable fetch utility. Three
   modes: zero-arg (Phase-1 default set), `--doc <law_id> <mst>`,
   `--search <query>`. Idempotent (skips existing files unless
   `--force`); atomic (tmp + rename, sync at end); single
   dependency (`curl`).

   Document fetches write to `data/raw/{law_id}/{mst}.xml` per
   ADR-011; search fetches write to `docs/api-samples/` (per
   ADR-011 "Out of scope" §2).

   Live-tested end-to-end on Fedora against the 법제처 OpenAPI:

   | Check | Result |
   |-------|--------|
   | Phase-1 fetch (2 docs + 1 search) | 2 wrote, 1 skipped (search existed) |
   | Well-formedness (`<?xml` declaration) | 3/3 OK |
   | SHA-256 content-hash (ADR-011 §5) | computed; will land on `legal_documents.content_hash` at ingest |
   | Idempotency re-run | all-skip, zero API calls |
   | `--force` overwrite | re-fetched, identical 42349 B |
   | `data/raw/` excluded from git status | confirmed |

   Sizes vs prior 2026-04-25 samples differ by ~1 KB (Act
   42349 vs 41402; Decree 97143 vs 94936) — consistent with
   API response variance over time (echoed metadata, possible
   minor re-publish), not a script bug. Commit `bdec007`.

## Patterns worth flagging

1. **First code session ships alongside a closing decision.** The
   project pattern through 2026-04-29 was decision-only. Today's
   shape — one ADR landing, one script landing — is a clean
   transition signal. The fetch script is small enough that it
   doesn't pull on §5b (no design ambiguity worth an ADR), but
   it's the first artefact under `src/`-adjacent territory.
   Subsequent sessions will be implementation-heavier.
2. **Decision freeze (ADR-010) makes scoping easier, not harder.**
   Open questions today were "where do raw responses live" and
   "how do we fetch them" — both answerable without re-opening
   any of the 10 frozen decisions. The freeze concentrated the
   decision surface to ingestion-side concerns.
