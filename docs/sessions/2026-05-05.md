## Follow-up — annex attachment binary retention

Seheon requested downloading annex files into:

```text
./data/annexes/{law_id}/{mst}/{annex_key}/{filename}
```

This crosses ADR-014's explicit out-of-scope boundary for binary download
and retention, so drafted
[ADR-016](../decisions/ADR-016-annex-attachment-binary-retention.md) as
`Proposed`, then accepted it after review.

Recommendation in ADR-016:

- store actual downloaded files under
  `data/annexes/{law_id}/{mst}/{annex_key}/{source_filename}`
- store repo-relative paths in `annex_attachments.stored_file_path`
- use the OC key as a request credential when needed, but never persist
  OC-bearing URLs in `source_attachment_url`
- download HWP/PDF rows with explicit `source_attachment_url` first
- review downloaded HWP/PDF files; they may already contain the relevant
  image/table renderings
- download image rows only after a verified image URL discovery pass
- if image files are not available in the expected law.go.kr path, return
  to ADR-016 and re-evaluate before trying another discovery method
- compute `checksum_sha256` and set `fetched_at` only after the file is
  actually written

Implementation proceeding under accepted ADR-016.

## Follow-up — PDF-default binary retention with HWP fallback

Seheon reviewed the retained HWP/PDF/GIF artifacts and concluded the
formats are redundant for Phase 1; PDF should be the default retained
format, with HWP as fallback.

Drafted
[ADR-017](../decisions/ADR-017-annex-pdf-default-hwp-fallback-binary-retention.md)
as `Proposed`; Seheon approved it and status was changed to `Accepted`.

Recommendation in ADR-017:

- keep all `annex_attachments` rows as source provenance
- make PDF the default local binary retention format
- fallback to HWP only when PDF retention is unavailable or invalid
- change downloader default to PDF-first with HWP fallback
- preserve explicit image download capability for future debugging.

ADR-017 no longer carries the operational dev DB cleanup procedure.
