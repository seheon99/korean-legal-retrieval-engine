# Session 2026-05-05 — annex ingestion + binary retention (ADR-015/016/017/018)

## Decisions

| ID | Decision |
|----|----------|
| ADR-015 | Migration management. Ordered raw PostgreSQL SQL files under `migrations/` plus a `schema_migrations` ledger. ADR-013 rename ships as a separate non-additive migration. |
| ADR-016 | Annex attachment binary retention. Files under `data/annexes/{law_id}/{mst}/{annex_key}/{source_filename}`; `annex_attachments.stored_file_path` holds the repo-relative path. OC key may be a request credential, never persisted in `source_attachment_url`. Compute `checksum_sha256` and `fetched_at` only after the file is written. |
| ADR-017 | PDF-default binary retention with HWP fallback. All `annex_attachments` rows preserved as source provenance. Image rows remain provenance-only (kept for future debugging). Default downloader = PDF-first with HWP fallback. |
| ADR-018 | Annex `content_text` de-layout normalization. Amends ADR-014's layout-preserving rule. `annexes.content_text` is parser-owned semantic text for chunking/retrieval; raw XML and retained PDFs are the fidelity sources. Local line-boundary classification: `JOIN_NO_SPACE`, `JOIN_WITH_SPACE`, `KEEP_BLOCK_BREAK`, `LOW_CONFIDENCE_REVIEW`. Structural boundaries by rule before tokenizer logic. Phase-1 substitute is a reviewed deterministic boundary classifier (no Korean tokenizer dep yet). |

## Empirical findings worth preserving

- ADR-014 image-discovery: rendered law.go.kr annex HTML exposes image
  `flDownload.do` URLs in document order. ADR-016 was amended to allow
  rendered-order image matching only when the rendered image URL count
  exactly matches the XML image row count for the annex.
- HWP/PDF artifact review: PDFs carry rendered document pages/text, not
  GIF files as embedded image objects. PDF page counts per Decree annex
  (`2,2,4,2,1`) sum to 11, matching the GIF image filename row counts.
  PDFs cover the rendered annex surface needed for Phase 1; HWP and GIF
  are redundant.
- ADR-018: the line-split symptom (`거\n리`) was present in upstream
  `<별표내용>` CDATA layout lines, not introduced by the parser.
  Classifier targets unambiguous repairs (`거`+`리` → `거리`); avoids
  unresolved domain-spacing claims like `복부 산통` vs `복부산통`.
