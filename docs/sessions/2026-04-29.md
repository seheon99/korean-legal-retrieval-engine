2. **TODO-10 → ADR-009 accepted (with 5 patches between Proposed
   and Accepted)** — `parent_doc_id BIGINT NULL REFERENCES
   legal_documents(doc_id) ON DELETE RESTRICT` self-FK on
   `legal_documents`. Linkage shape: 1:N tree (Act ↔ Decree ↔
   Rules), not M:N graph. Empirical basis: the API exposes no
   explicit parent-pointer field; title-pattern matching
   (`{Act_title} 시행령`) is the deterministic signal.

   Initial Proposed draft drew a 5-point critique:
   (1) **soft-enforcement reasoning weak** — adopted asymmetric
   CHECK `chk_legal_documents_act_no_parent CHECK (doc_type <>
   '법률' OR parent_doc_id IS NULL)`;
   (2) **title-lookup uniqueness not structurally guaranteed** —
   added partial UNIQUE INDEX `ux_legal_documents_current_act_title
   ON (title) WHERE doc_type = '법률' AND is_current = TRUE`;
   (3) **reverse-traversal index missing from Consequences** —
   added `ix_legal_documents_parent ON (parent_doc_id) WHERE
   parent_doc_id IS NOT NULL`;
   (4) **`ON DELETE SET NULL` not justified** — switched to RESTRICT
   with explicit comparison vs CASCADE (Decrees outlive Acts) and
   SET NULL (silent orphan);
   (5) **Rules-parent heuristic oversimplified** — 부령 제1조
   typically reads "이 규칙은 「○○법」 및 같은 법 시행령에서
   위임된 사항…" (delegated by both Act and Decree), so single-
   line title-strip pattern doesn't apply; preview dropped,
   assignment deferred until first 시행규칙-bearing statute enters
   scope.

   All 5 patches applied, then approved. First ADR in the broader
   ADR-004→010 arc to draw structural critique — informs pattern
   #2 below. Commit `fa42376`.

## Patterns worth flagging

1. **Empirical correction can come from agent slip too.** During
   ADR-008 drafting, an Explore subagent's analysis misclassified
   항/호/목 as unmapped XML and briefly suggested "JSONB
   justified for this volume." Caught by re-reading ADR-006's
   level mapping. Generalization of the prior session's pattern
   #1 (re-verify against ERD-as-of-now, not against memory):
   subagent output is still tool output, not ground truth.
   Re-verify against ERD-as-of-now, not against memory *or*
   against agent summaries.
2. **Devil's Advocation streak broke at the first structural
   ADR.** ADR-004→007 each approved with light refinements; no
   full rewrites. ADR-008 also approved cleanly. ADR-009's first
   draft drew a 5-point critique with mandatory patches before
   approval. Distinguishing feature: ADR-004→008 were
   representational (closed-taxonomy column choice, sibling-vs-
   sidecar capture, separate-table-vs-merge, JSONB-or-not);
   ADR-009 was structural (multi-doc-type linkage with cross-
   table indexing and population-rule consequences). Predicts:
   TODO-5 (amendment-tracking temporality across `node_id`) will
   draw similar structural fire when its time comes — also
   cross-table, also affects ingestion ordering, also has a
   population-rule shape.
3. **Docker dry-run is the project pattern for migration syntax
   checks.** No local `psql` installed; `postgres:16-alpine` with
   `BEGIN; \i migration.sql; ROLLBACK;` confirms tables, indexes,
   and constraints in seconds with zero state persisted. Worth
   keeping as the standard pre-commit verification step for
   future migrations (`002_*.sql` and onward).
4. **A Phase-1 freeze cluster has a recognizable shape.** Three
   ADRs in 90 minutes is well above the project's typical
   one-ADR-per-session cadence. Reading: ADR-008 was already
   the "obvious next" item per the prior session's log,
   ADR-009 was the only remaining ERD-shape decision before a
   freeze was justifiable, and ADR-010 was the freeze itself.
   Cluster shape: **last-representational + last-structural +
   freeze**. Worth recognizing if a similar end-of-phase moment
   appears in future ERDs (사법판단, 유권해석, etc.).

   constraint blocks ingestion and the verification trigger
