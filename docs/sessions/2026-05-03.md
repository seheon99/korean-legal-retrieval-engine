## Code landed

**Load layer** under [src/ingest/](../../src/ingest/), naming follows
CLAUDE.md §4 "ingestion-pipeline" (not RAGFlow's "Load"; `load` collides
with verb usage and `ingest` is more conventional).

- `__init__.py`, `__main__.py`, `records.py` (frozen pydantic
  `Document`), `parse.py` (`discover()`, `parse_doc()`, `sha256_file()`),
  `populate.py` (three-phase orchestrator
  `PHASE_ORDER = ('법률', '대통령령', '총리령', '부령')`, ADR-009
  `_resolve_parent()` with fail-fast on Decree-without-Act, one
  transaction per phase so Phase N's COMMIT is visible to Phase N+1's
  parent lookup).

**Idempotent re-ingest** — `_skip_if_present(conn, doc)` SELECTs by `mst`:

- no row → INSERT
- match on `content_hash` → skip
- mismatch → raise `ContentMismatchError` (auto-supersede deferred to
  amendment-tracking ADR; loud failure is the right Phase-1 floor)

**Docker dev stack** — [docker-compose.yml](../../docker-compose.yml):
`database` service (`pgvector/pgvector:pg16`, picked over
`postgres:16-alpine` so the dev image stays stable when chunks land;
migrations bind-mounted to `/docker-entrypoint-initdb.d:ro`); `ingest`
one-shot service (python:3.11-slim + `psycopg[binary]` + `pydantic`,
bind-mounts source for dev comfort). Service named `database` (not `db`)
per house no-abbrev style.

**Compose hardening** —

- Credentials sourced strictly from `.env` (`${POSTGRES_USER:?…}` form
  fails compose-config render if `.env` doesn't supply them).
- Healthcheck uses `$${VAR}` (escaped) so the in-container shell resolves
  from its own env rather than compose substituting credentials into the
  rendered config.
- Password moved out of the DSN — `DATABASE_URL` carries
  `user@host:port/dbname`; `PGPASSWORD` is a separate env var that
  libpq/psycopg read automatically. Avoids URL-encoding pain on special
  characters in the password.
- `requirements.txt` relocated to
  [docker/ingest-requirements.txt](../../docker/ingest-requirements.txt)
  next to the only consumer (`docker/ingest.Dockerfile`); leaves room
  for `docker/retrieval-requirements.txt` later.

## Empirical findings worth preserving

- 형법 `mst=284025` exposed 49 `<조문가지번호>` instances (가지번호 ∈ {2, 3}),
  encoded directly into `조문키` as `xxxx{BB}{T}` 7-digit pattern. First
  N=1 sample looked clean.
- 도로교통법 `mst=281875` exposed 37 `<호가지번호>` instances (max 가지번호
  = 5), structurally distinct: sibling `<호>` with same `<호번호>` plus
  extra `<호가지번호>`; parser composes the key (vs API providing it for
  조). Zero `<항가지번호>` / `<목가지번호>` / level-1..4 가지번호 in either
  sample.
- Implicit `<항>` (bare `<항>` no `<항번호>`) confirmed in SAPA Act §2.

## Project conventions / lessons

- **Sample breadth matters when an ADR claims "settled by empirical
  inspection."** ADR-012 went deferred → 조-only → 조+호 → tagless across
  four scope corrections. Pull at least two structurally-different
  statutes before flipping such ADRs to Accepted; one sample falsifies
  but never confirms scope.
- **Verification-trigger pattern:** when an ADR commits to an
  API-contract assumption (e.g., "조문키 always has 7 digits"), it should
  ship with a parser-side assertion. Halt on violation, not warn.
- **Phase-2 follow-up vs premature ADR.** ADR-012 records the *intent*
  to drop `sort_key`; the formal *drop* decision waits for the Phase-1 →
  Phase-2 boundary when the full Phase-2 context is available.
- **Parser-implementation work surfaces gaps that pure design review can
  miss.** The 호-branch encoding gap appeared during parsing, not during
  ADR-001 through ADR-011 design.
