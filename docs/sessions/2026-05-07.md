## Project goal review

Current position:

> This project is a production-grade Korean legal retrieval engine, not a
> chatbot and not a full RAG system.

The project's durable goal is to prove that a direct, measurable retrieval
engine can outperform or complement thin 법제처 API wrappers by:

- preserving source-native legal structure in separate source ERDs;
- building a unified `chunks` search layer with real source FKs;
- supporting authority, provenance, license, and temporal metadata;
- quantitatively evaluating Korean legal retrieval quality with Recall@K
  and MRR;
- comparing hybrid retrieval behavior against weaker or narrower baselines;
- keeping generation, multi-agent routing, and orchestration frameworks out
  until retrieval quality is measured.

Phase-1 goal:

> Build the SAPA + OSH statutory baseline from canonical `eflaw` source
> rows through the first hybrid retrieval evaluation.

Phase-1 is no longer just an ingestion walking skeleton. The source layer is
now populated enough that the next meaningful proof point is retrieval:

- generate chunks from `structure_nodes` and `annexes`;
- default retrieval/eval to the legally effective slice as of
  `2026-05-06`;
- require explicit query mode for head/future-effective slices;
- implement BM25 + vector candidate retrieval;
- fuse with RRF, giving the original query 2x weight;
- rerank with `bge-reranker-v2-m3`;
- compare `bge-m3` and KURE before freezing the embedding model.

Out of scope remains explicit:

- no generation layer;
- no LangChain or LlamaIndex;
- no non-statutory Phase-1 expansion before the statutory baseline is
  measured;
- no LLM router or multi-agent workflow before an evaluation baseline
  exists.

## Follow-up — Retrieval operation model

Seheon introduced a retrieval operation model for the first public contract.

Core direction:

- The primary consumer is an AI agent, not a human search UI.
- The engine should expose legal-intent operations, not raw retrieval knobs.
- Deterministic retrieval and hybrid evidence retrieval are separate
  operation classes.
- Deterministic retrieval resolves known legal citations or identifiers to
  exact effective provisions. It is identity-based; BM25 or semantic search
  may only be fallback normalization, never final authority.
- Hybrid evidence retrieval handles concept-level legal queries using
  lexical retrieval, semantic retrieval, fusion, reranking, structured
  filters, and provenance packaging.
- Hybrid evidence retrieval does not generate an answer. It returns
  citation-ready evidence objects for a downstream agent.
- Every valid result must include citation, provenance, source identity,
  effective-date metadata, and hierarchy context.

This is API-contract-level direction, so it falls under the ADR-first rule.

Created:

- [docs/decisions/ADR-021-retrieval-operation-contract.md](../decisions/ADR-021-retrieval-operation-contract.md)
  with status `Proposed`.

No retrieval schema, code, API route, or config change should happen until
ADR-021 is reviewed and accepted or revised.

## Follow-up — Graph modeling roadmap

Seheon introduced the KLRE graph modeling roadmap direction.

Core direction:

- KLRE should become graph-ready, but the current ERD remains canonical
  normalized storage.
- The graph is a derived projection layer, not a replacement for the ERD
  and not an independent source of truth.
- The graph roadmap proceeds from structure to meaning:
  Layer 1 structural graph -> Layer 2 reference graph -> Layer 3
  definition graph -> Layer 4 legal semantic graph.

  names, schemas, indexes, migrations, projection jobs, and extraction
  logic are deferred.

  graph projection from the current statute ERD.

  relationships are not Layer 1 containment.
- Article branches are ordering issues, not graph relationship types.
- Deleted provisions are lifecycle/status or projection-policy issues, not
  graph relationship types.
- Supplementary provisions, annexes, and forms belong in Layer 1 with
  Phase-1 simplifications.
